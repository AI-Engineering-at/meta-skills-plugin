## 5a. Branch-Chaos Root-Cause-Analyse + Worktree-Recommendation

**Observed** (phantom-ai working tree during session end):
- 5 uncommitted modifications across 3 unrelated topics
- 3 untracked directories for separate features
- Everyone working in one folder → everyone in one branch → gegenseitige Blockade

**Root cause**: Git's default single-working-tree model meets multi-terminal workflow. 4 Terminals am gleichen Ordner = 4 Mal derselbe Staging-Area.

**Recommended fix**: Git Worktrees (pro Topic ein Working-Tree, pro Working-Tree ein Branch).

```bash
# Einmal-Setup (ca. 5 min):
cd ~/Documents/phantom-ai
git worktree add ../phantom-ai.persona feature/persona-update
git worktree add ../phantom-ai.meta-skills-ops chore/meta-skills-bumps
git worktree add ../phantom-ai.signal-bridge feature/signal-bridge
git worktree add ../phantom-ai.experiments experiments  # catch-all für WIP

# Dann pro Terminal:
# Terminal 1: cd ~/Documents/phantom-ai.persona         → main-Branch isoliert
# Terminal 2: cd ~/Documents/phantom-ai.meta-skills-ops → chore Branch isoliert
# usw.
```

**Vorteile:**
- Keine branch-guard-Notwendigkeit — physisch unmöglich Topics zu mixen
- Keine `git stash`-Gymnastics (stash war in Vorsessions Quelle von Konflikten)
- Jeder PR ist sauber auf 1 Topic
- Joe kann parallel arbeiten OHNE mental-switch
- Rule 22 (Task → Branch → PR) wird plötzlich natürlich zu folgen

**Gefahren / Gotchas:**
- Submodule (meta-skills, comfyui-build): `git worktree add` teilt `.git` aber Submodule müssen pro Worktree separat init'd werden → 1-time cost
- .gitignore muss `../phantom-ai.*` nicht ignorieren (sind außerhalb des Repos — convention schon safe)
- Deployment-Scripts die hardcoded `cd ~/phantom-ai` nutzen → pro worktree invariant halten oder explizit `cd <worktree>` setzen

**Migration-Path** (inkrementell, kein Big-Bang):
1. **Session 1** (30 min): Haupt-Repo bleibt, 1 Worktree erstellen für das nächste Feature. Lerne Pattern. Commit tests.
2. **Session 2** (60 min): 3-4 Worktrees für laufende parallele Streams. Dokumentieren in Rule 22.
3. **Session 3** (30 min): `scripts/setup-worktrees.sh` schreiben für neue Maschinen / Onboarding.
4. **Session 4+**: branch-guard EXPLIZIT entfernen (`git rm .githooks/branch-guard.sh` + commit), weil strukturell nicht mehr nötig.

**Falls nicht Worktrees**: Alternative wäre disziplinierte Branch-Erstellung pro Task (Rule 22 strikt), aber das hat Joe als zu schwerfällig abgelehnt. Worktrees = zero-friction Branch-Pflicht.
